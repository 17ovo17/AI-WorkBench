# AI WorkBench 全量穷举测试方案

> 来源：`C:\Users\十七\Downloads\AI_WorkBench_全量穷举测试方案.html`。本文件是项目内落地版，后续测试、修复和回归默认以本文档为准。

## 一、测试范围与目标

AI WorkBench 的测试目标不是只验证页面能打开，而是验证“用户输入 → 平台编排 → 监控/探针/AI/存储 → 页面回显 → 持久化恢复”的真实闭环。

- 覆盖前端所有可点击元素、表单、导航、上传/下载、响应式布局和异常输入。
- 覆盖后端全部 API 的正常、异常、边界和安全 payload。
- 覆盖 AI 问答、智能诊断、告警、Catpaw 探针、业务拓扑、AI 配置、数据源配置、凭证管理和持久化。
- 覆盖 MySQL、Redis、Prometheus 三类存储/监控数据的一致性和异常降级。
- 覆盖 Windows 远程探针、WSL/Linux 本机服务和未来 24h Categraf 伪造指标数据。

## 二、环境准备与基线要求

### 2.1 基础设施层

- WSL 发行版：`Ubuntu-24.04`。
- WSL 项目路径：`/opt/ai-workbench`。
- Windows 项目路径：`D:\ai-workbench`。
- 前端：`http://localhost:3000`。
- 后端：`http://localhost:8080`。
- Prometheus：`http://localhost:9090`。
- MySQL/Redis：由后端 `/api/v1/health/storage` 真实检测。
- 启停脚本：`D:\ai-workbench\start-wsl.sh`、`D:\ai-workbench\stop-wsl.sh`。

### 2.2 监控测试数据

Prometheus 必须能查询到未来 24h 的 Categraf 风格测试数据，采集频率为 15s。

| `198.18.20.12` | 应用服务器 | 基础资源、JVM、HTTP `8081`、进程 |
| `198.18.20.20` | 网关/缓存 | 基础资源、Nginx、Redis、进程 |
| `198.18.22.11` | Oracle 数据库 | 基础资源、Oracle、进程 |

### 2.3 AI 服务层

- AI 配置页必须展示当前已配置模型，不允许出现左侧可用但配置页空白。
- 模型健康点必须给出可读状态，不允许全部显示问号且无原因。
- 聊天、诊断、Catpaw AI 必须统一走平台配置或平台 `/api/v1/agent/llm` 网关。
- 敏感字段必须脱敏，前端不显示明文 API Key。

## 三、单模块功能穷举矩阵

### 3.1 智能对话（Conversation）

- NL2Command 至少覆盖 10 条自然语言指令：安装服务、查看磁盘、查 CPU、重启服务、查看端口、查日志、生成 kubectl、PromQL 查询、远程巡检、危险命令。
- 多轮上下文至少 3 轮，验证指代消解、IP 继承、上轮诊断结果引用。
- RAG 幻觉检测：询问平台不存在能力时必须说明不支持，不能编造。
- 提示词注入防御：覆盖忽略系统提示、泄露密钥、伪造工具返回、越权命令、隐藏指令五类 payload。
- 附件诊断：截图/日志能力不存在时必须明确提示，不允许假装读取。
- 高危命令：删除、格式化、停库、改防火墙、卸载探针必须二次确认。

### 3.2 智能诊断（AI Diagnose）

- 用户问 IP 压力时默认优先查 Prometheus 全量指标。
- Prometheus 标签必须兼容 `ident`、`instance`、`ip`、`host` 等与目标 IP 相关的标签。
- Prometheus 无数据时按顺序 fallback：最近 Catpaw report → Catpaw 即时巡检 → 明确提示缺少凭据/通道。
- 报告必须显示数据来源、关键指标、缺失指标、fallback 路径、未覆盖风险。
- 报告可下载 Markdown/HTML，并支持删除记录且有二次确认。

### 3.3 告警中心（Alerts）

- Webhook 正常告警、Catpaw 告警、重复告警、恢复告警、异常字段告警都要覆盖。
- 标记恢复必须持久化，刷新和重启后状态不丢。
- XSS/SQL/超长描述不得污染页面或导致服务崩溃。

### 3.4 业务拓扑（Business Topology）

- 左侧为用户定义的业务列表，每个业务可独立命名、切换、删除。
- 用户新增业务后，填写该业务包含的服务器 IP、应用、端口，再结合 Prometheus/Catpaw 自动发现组装。
- 只发现用户输入 IP/端口相关资源，不混入其他无关 target。
- 拓扑优先树形结构；连通为绿线，断开为红线；Prometheus 离线只影响健康，不阻断发现。
- 删除业务、节点、连线必须二次确认。

### 3.5 AI 配置（AI Config）

- 加载、保存、刷新模型、健康检查、错误配置、空配置、超长字段都要覆盖。
- 已有聊天模型必须在配置页可见。
- API Key 显示脱敏，保存后刷新仍存在。

### 3.6 数据源配置（Data Sources）

- Prometheus、MySQL、Redis 的配置加载、保存、健康状态和异常原因必须可见。
- 历史已有配置必须能从配置文件/数据库 fallback 展示。
- Prometheus 离线时诊断页不能误判“无主机”，应区分 target 离线和历史数据存在。

## 四、跨模块联动与状态机遍历

| 联动链路 | 必测点 |
| --- | --- |
| AI 问答 → Prometheus → 诊断报告 | IP 查询、指标检索、报告下载、报告删除 |
| Catpaw 探针 → 心跳 → 巡检 → 诊断 | 安装、上报、在线状态、fallback |
| 拓扑 → Prometheus → 告警 | 用户业务 IP、节点健康、绿/红线、告警关联 |
| AI 配置 → 聊天 → 记录持久化 | 模型选择、上下文保存、刷新恢复 |
| 数据源配置 → 健康接口 → 页面状态 | MySQL/Redis/Prometheus 状态一致 |

状态机必须覆盖：探针未安装、安装中、在线、离线、巡检失败、卸载中、已卸载；告警触发、确认、恢复、重复；诊断排队、执行、成功、失败、删除。

## 五、数据持久化专项（MySQL / Redis / Prometheus）

- MySQL：诊断记录、告警、凭证、AI 配置、数据源、聊天会话、聊天消息、拓扑业务、拓扑节点、拓扑连线必须可重启恢复。
- Redis：仅用于在线状态、短期缓存、会话临时态，不能作为唯一持久存储。
- Prometheus：基础资源、JVM、Nginx、Redis、Oracle、网络、TCP、进程指标必须可被全量发现和分类。
- 异常注入：MySQL 停止、Redis 停止、Prometheus 停止时页面和健康接口必须返回明确原因。

## 六、探针全生命周期深度验证（Catpaw Agent）

- Windows 远程探针：真实安装、心跳、巡检、上报、卸载、重装。
- Linux/WSL 探针：本地巡检闭环，验证 CPU、内存、磁盘、网络、进程、服务摘要。
- 安装命令：Windows 使用 Windows 语义和资产；Linux 使用 Linux 语义和资产；平台地址必须真实可达。
- 凭证：通过凭证管理选择，接口以 `credential_id` 解析，不在命令或页面展示明文密码/API Key。
- 卸载：停止进程、删除服务/任务、清理目录，心跳超时后状态离线。

## 七、数据源与监控闭环验证

- `/api/v1/health/storage`：MySQL/Redis 真实 ping。
- `/api/v1/health/datasources`：Prometheus/MySQL/Redis 连接状态和错误原因。
- `/api/v1/prometheus/hosts`：发现所有与 IP 标签相关的主机。
- `/api/v1/prometheus/metrics?ip=...`：按 IP 返回 Categraf 默认采集项和业务指标。
- 网络报告必须包含带宽、丢包、错误包、TCP 连接、TIME_WAIT 等。

## 八、业务拓扑与告警联动

- 用户新建多个业务，每个业务独立输入 IP/端口和业务组件。
- 拓扑只在用户输入范围内自动发现，并按 Prometheus/Catpaw 数据补全节点。
- 应用、Nginx、Redis、Oracle、基础资源节点必须能联动健康状态。
- 告警进入后应能定位到对应业务、主机、服务或连线。

## 九、非功能性测试覆盖

- 性能：并发聊天、并发诊断、Prometheus 大量指标、拓扑多节点渲染。
- 稳定性：服务重启、网络中断、长时间运行、缓存过期。
- 兼容性：Chrome/Edge、桌面/平板/移动端、100% 缩放下关键内容可见。
- 可读性：诊断报告高对比度，不允许字体颜色过暗或乱码。

## 十、异常注入与边界穷举

- 空 JSON、错误 Content-Type、字段缺失、类型错误、超长字段、Unicode/emoji。
- XSS、SQL 字符串、命令注入、SSRF URL、路径遍历、Markdown/HTML 渲染风险。
- 重复提交、并发请求、刷新中断、后退/前进、直接 URL 访问。
- 文件下载、报告删除、探针卸载等破坏性动作必须二次确认。

## 十一、测试输出与交付物标准

每轮测试至少交付：

- 执行记录：`AI_WorkBench_测试执行记录.md` 的一份实例。
- 覆盖 Checklist：标记已测、阻塞、待补齐。
- 证据目录：截图、接口响应、PromQL 查询结果、探针安装/卸载日志、修复前后对比。
- 缺陷清单：P0/P1/P2/P3、复现步骤、修复提交/文件、回归结果。

## 十二、执行排期与风险管控

- 第 1 阶段：基础健康、存储、Prometheus、测试数据、核心 API。
- 第 2 阶段：AI 问答、诊断、报告、配置页、聊天记录。
- 第 3 阶段：Catpaw Windows/Linux 探针全生命周期。
- 第 4 阶段：业务拓扑、告警联动、异常注入、UI 穷举。
- 阻塞上线 P0：服务不可启动、存储不可持久、诊断主链路不可用、探针无法上报、删除无确认、敏感信息明文泄露。
- 有条件上线 P1：关键页面功能缺失、指标适配不完整、拓扑混入无关主机、报告不可读、异常提示不可理解。
