|\n\n### 1.2 网络与连接诊断\n- **Prometheus**：`net_bits_recv/send`, `net_drop_in/out`, `tcp_retrans`, `tcp_time_wait`\n- **Catpaw**：网卡速率协商状态（是否降速到 100Mbps）、防火墙规则、DNS 解析状态\n- **判定**：\n  - retrans > 1% 且 drop 增长 → 网络质量差或拥塞\n  - tcp_time_wait > 10000 → 短连接风暴或端口耗尽风险\n  - Catpaw 报网卡协商降速 → 硬件/线缆故障\n\n### 1.3 进程与服务诊断\n- **Prometheus**：`procstat_cpu_usage`, `procstat_mem_rss`, `procstat_num_fds`\n- **Catpaw**：关键进程存活、systemd 状态、文件句柄限制（ulimit）\n- **判定**：\n  - 进程 CPU 突增 10 倍 → 死循环或 GC 异常\n  - num_fds 接近 ulimit → 句柄泄漏\n  - Catpaw 报进程不存在但 Prometheus 有指标 → 僵尸指标或采集延迟\n\n### 1.4 单机健康评分\n每台机器输出百分制评分：\n- 90-100：健康\n- 70-89：亚健康（存在 1-2 个 P2 级问题）\n- 50-69：病态（存在 P1 级问题或多项 P2）\n- 0-49：危险（存在 P0 级问题，需立即介入）\n\n---\n\n## 第二层：集群/服务资源判断（Cluster Resource Analysis）\n基于 `app`/`service`/`cluster` 标签聚合多台机器，做资源层判断。\n\n### 2.1 容量与饱和度分析\n- **CPU 容量**：`avg(cpu_usage_active) by (service)` + `max(cpu_usage_active) by (instance)`\n  - 均值 > 60% 且峰值 > 90% → 容量不足，需扩容\n  - 均值 < 20% 但实例数 > 3 → 过度配置，可缩容\n- **内存容量**：`avg(mem_used_percent) by (service)`\n  - 持续上升趋势（7d 对比）→ 内存泄漏或业务增长\n- **磁盘容量**：`max(disk_used_percent) by (instance)` \n  - 单点磁盘 > 85% 且该服务其他节点 < 50% → 数据倾斜\n- **连接池/线程池**：如有业务暴露指标（如 `http_requests_active`），判断池化资源饱和度\n\n### 2.2 负载均衡与数据倾斜\n- 同服务下各节点 CPU 使用率标准差 > 20% → 负载不均（需检查 LB 策略或热点数据）\n- 同服务下某节点流量是其他节点 3 倍以上 → 存在\"热点机器\"\n\n### 2.3 依赖与链路风险\n- **P0（致命）**：单点故障、数据丢失风险、安全漏洞、容量耗尽（< 7 天）\n2. **P1（高危）**：性能瓶颈导致用户体验下降、主从延迟过大、核心依赖不可用\n3. **P2（中危）**：资源浪费、监控盲区、配置不合理\n4. **P3（低危）**：优化建议、长期技术债\n\n### 3.3 资源判断结论\n- **当前资源利用率**：CPU / 内存 / 磁盘 / 网络的集群均值与峰值\n- **瓶颈预测**：基于近 7 天/30 天趋势，预测何时触及容量上限（线性回归或拐点检测）\n- **成本评估**：识别低利用率机器（CPU < 10% 持续 7 天）、过度配置实例、闲置存储\n\n### 3.4 巡检行动建议\n- **立即执行（0-24h）**：P0 问题处理\n- **本周完成（1-7d）**：P1 问题修复 + 容量扩容\n-
